PART I — PERFORMANCE OPTIMIZATION

Current Codebase Bottlenecks PERFORMANCE

Critical issues found through deep analysis of the engine, scanner, and backend packages.

CRITICAL   O(n²) Bubble Sort in Scanner

File: internal/scanner/scanner_windows.go:404-427

sortPathsByDepth() uses a bubble sort to order files by depth. For 1 million files, this requires ~1012 comparisons. Replacing with sort.Slice (O(n log n)) would reduce this from hours to seconds.

Impact
Catastrophic at scale

HIGH   Mutex Lock Per File Deletion

File: internal/backend/windows_advanced.go:737-768

Stats tracking uses sync.Mutex for incrementAttempt() / incrementSuccess() — 3+ mutex operations per file. Should use sync/atomic as per project conventions.

HIGH   Mutex-Protected Append in Scanner

File: internal/scanner/scanner_windows.go:290-292

Every file found locks pathInfosLock to append to a shared slice. With parallel workers, this becomes a serialization point. Pre-allocating and using per-worker local buffers with a final merge would eliminate contention.

MEDIUM   Depth Calculation Bug

File: internal/scanner/scanner_windows.go:239

filepath.SplitList() splits by PATH separator (; on Windows), not path separator (\). Should count os.PathSeparator occurrences instead.

MEDIUM   Unoptimized FindFirstFile

File: internal/scanner/scanner_windows.go:227

Uses FindFirstFile instead of FindFirstFileEx with FindExInfoBasic + FIND_FIRST_EX_LARGE_FETCH. The optimized version uses 64KB buffers (vs 4KB) and skips 8.3 name retrieval, yielding 2-6x scanning speedup.

MEDIUM   filepath.Join Overhead

File: internal/scanner/scanner_windows.go:261

filepath.Join calls filepath.Clean internally, causing 2 allocations per file. For known-clean paths from directory enumeration, direct concatenation (dir + "\\" + name) is safe and avoids this overhead.

LOW   Zero-Capacity Slice Init

File: internal/scanner/scanner_windows.go:104-110

Slices initialized with make([]string, 0) instead of pre-allocated capacity. For million-file directories, this causes repeated growth reallocations under mutex lock.

Key Insight: Per-file syscall count (1-3) matters far less than I/O pipeline saturation. Each syscall costs ~1-2µs, but filesystem journal operations cost 10-100µs on SSD. For 1M files, switching from 3 to 1 syscall saves ~2 seconds, while parallelism can save minutes. The real win is keeping many operations in-flight simultaneously (IOCP pattern used by byenow).

Already Implemented

  • NtDeleteFile — single-syscall deletion (engine.go)
  • FileDispositionInfoEx with POSIX semantics
  • Fallback chain: InfoEx → Info → DELETE_ON_CLOSE → NtDeleteFile → Win32
  • UTF-16 pre-conversion during scan phase

Potential Improvements

  • IOCP-based dispatch — keep N async ops in-flight vs blocking goroutines
  • FILE_OPEN_REPARSE_POINT flag to skip reparse point resolution
  • Detect NtfsDisableLastAccessUpdate and recommend enabling it
  • Move-then-delete pattern — rename to temp, delete in background

Linux: io_uring Batch Deletion PERFORMANCE

Since Linux kernel 5.11, IORING_OP_UNLINKAT enables true batched deletion.

How It Works

  • Prepare multiple SQEs (Submission Queue Entries), each an unlink operation
  • Submit all in a single io_uring_enter() syscall
  • Kernel processes asynchronously, posts completions to CQ
  • Amortizes syscall overhead across N operations
  • Optimal batch size: 64-256 operations

Performance Characteristics

  • Single-operation io_uring is slower than plain unlink()
  • Benefit comes from batching — one syscall for N deletions
  • Up to 30% less CPU utilization with large batches
  • Kernel can reorder and parallelize filesystem ops internally
Recommendation: Create backend_linux.go analogous to windows_advanced.go, using io_uring with IORING_OP_UNLINKAT in batches of 64-256. Go libraries: iceber/iouring-go or direct syscalls via cgo. The current GenericBackend uses simple os.Remove() and misses this optimization entirely.

Critical: d_type Preservation

Go's os.ReadDir() calls getdents64 but discards the d_type field (file type from directory entry), then makes additional stat() calls. godirwalk preserves d_type, eliminating one syscall per entry.

Buffer Size Matters

Go's os.ReadDir() uses 8KB buffer for getdents64; glibc uses 32KB. FIND_FIRST_EX_LARGE_FETCH on Windows uses 64KB vs default 4KB. The WholeTomato team measured 50-200x speedup in file attribute queries with larger buffers.

Memory & Allocation Optimization PERFORMANCE

Reducing GC pressure for million-file workloads.

sync.Pool for Buffers

Reuse []byte buffers for path building and UTF-16 conversion. Eliminates per-file heap allocations that trigger GC.

Arena Allocation

Allocate all PathInfo structs in a region during scan, free entire arena after deletion. Eliminates GC scanning of millions of small objects.

Zero-Copy Strings

Since Go 1.20, unsafe.String and unsafe.Slice enable zero-allocation string/byte conversions for UTF-16 paths.
